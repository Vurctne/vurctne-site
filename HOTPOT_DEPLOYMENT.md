# /hotpot deployment — handoff (v6: per-investor passwords, no CF Access)

> Investor demo at `vurctne.com/hotpot`, password-only access (no email PIN).
> Each investor gets a unique password generated by the admin dashboard at `vurctne.com/hotpot/admin/`.
> Watermark + copy-event tracking per investor.
> Updated 2026-05-02.

## How it works now

```
You (admin)               Investor                Worker + D1
─────────                ──────────              ──────────────
Open /hotpot/admin/  →   (private to you)        validates ADMIN_PASSWORD
Add investor "Alice"
  (note: "Sequoia")
Server generates pw  ←   (returned once)         insert into investors
  e.g. xR8d-Kp42-Vmz9                            (stores hash + salt)
You share with Alice:
  "/hotpot URL + xR8d-Kp42-Vmz9"

                         Alice opens /hotpot
                         enters xR8d-Kp42-Vmz9 → POST /api/unlock
                                                 finds matching investor
                                                 returns Alice's name + bundle key
                         decrypts demo locally
                         demo loads with         logs unlock-success event
                         "Alice" watermark       (with investor_id)

                         Alice tries Ctrl+C  →   POST /api/copy-event
                                                 (with signed demo_token)
                                                 logs copy event with investor_id

You refresh dashboard ← shows Alice's row: Active, 1 visit, 1 suspicious event
                                                 → click "事件" to see details
```

If Alice's password leaks, click "撤销" on her row → her password stops working immediately. Other investors unaffected. Generate a new pw with "换密码".

## Threat model — v6

| Threat | Mitigated by |
|---|---|
| URL enumerated by random visitor | Demo password gate (server-side validated against D1) |
| Brute-force the demo password | 12 random chars from 50-char alphabet ≈ 2^67 keyspace; 200–400 ms artificial delay per fail; failed attempts logged to admin |
| Password leaked by one investor | Revoke that one investor → their pw stops working; others unaffected |
| At-rest leak (CDN / cache scrape) | Bundle is AES-256-GCM ciphertext; key is a Worker secret, never in repo |
| Investor saves the rendered demo | Decrypted demo is in iframe srcdoc; obfuscated JS; per-investor watermark with investor name |
| Investor copies content | Copy event logged with investor_id; visible in admin dashboard |
| Investor tampers with copy-event JS to fake another investor | Demo token is HMAC-signed with `ADMIN_SESSION_SECRET`; only Worker can mint; client can't forge a token for a different investor |
| Stolen admin cookie | 8h expiry; HMAC-signed; HttpOnly + Secure + SameSite=Strict |
| Admin password brute force | 250–500 ms artificial delay per fail |
| **Determined adversary with phone camera** | Watermark still attributes to a specific investor name |

## D1 schema reference

```sql
CREATE TABLE investors (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  name TEXT NOT NULL UNIQUE,
  note TEXT,
  password_hash TEXT NOT NULL,
  password_salt TEXT NOT NULL,
  added_at INTEGER NOT NULL,
  revoked_at INTEGER  -- NULL = active
);

CREATE TABLE events (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  ts INTEGER NOT NULL,
  investor_id INTEGER,  -- NULL for unattributed events (e.g., wrong-password attempts)
  action TEXT NOT NULL,
  detail TEXT,
  ip TEXT, country TEXT, ua TEXT, referer TEXT,
  FOREIGN KEY (investor_id) REFERENCES investors(id)
);
```

## Common operations

- **Add investor:** Admin dashboard → fill name + note → "添加 + 生成密码" → copy password → share with investor + URL
- **Rotate password:** "换密码" → confirm → new password → share with investor
- **Revoke / restore:** "撤销" / "恢复" buttons. No re-enabling needed if you just want to pause.
- **Delete:** "删除" — fully removes investor row. Their event records remain (for audit) but lose investor_name association.
- **View events for one investor:** Click "事件" → drawer shows chronological timeline.
- **See who's been guessing wrong passwords:** Admin dashboard's "24小时内的密码错误尝试" section.

## Rotation

**Admin password:** `npx wrangler secret put ADMIN_PASSWORD` with new value. Existing admin sessions remain until expiry; new logins use new pw.

**Force-invalidate all admin sessions AND demo tokens:** rotate `ADMIN_SESSION_SECRET`. (Demo tokens are HMAC'd with this too.) Investors will need to re-enter their pw after this.

**Bundle key:** rotate `BUNDLE_DECRYPT_KEY` AND re-encrypt the bundle with the new key. Tell me "rotate hotpot bundle key" — I regenerate everything and place a new `index.v2.html`.

**Per-investor passwords:** "换密码" button in admin dashboard. No deploy needed.

## Troubleshooting

**"server not configured" on /api/unlock**
→ One of `BUNDLE_DECRYPT_KEY` / `ADMIN_SESSION_SECRET` not set. Verify with `npx wrangler secret list`.

**Decryption sanity check fails after correct password**
→ `BUNDLE_DECRYPT_KEY` doesn't match the key used to encrypt the bundle. Rotate-key flow.

**Admin login returns 401 with right pw**
→ Worker hasn't redeployed since secret change. Push any commit to trigger a redeploy, or run `npx wrangler deploy`.

**Admin says "D1 binding not configured"**
→ wrangler.jsonc D1 binding hasn't taken effect. Check Settings → Bindings → D1 in dashboard, or `npx wrangler deploy`.

**Investor shows "未访问" but they say they unlocked**
→ Either (a) they used a wrong password and saw the error, or (b) they unlocked using a DIFFERENT investor's password. Check unlock-fail entries and the events drawer for unlock-success entries with their IP.
